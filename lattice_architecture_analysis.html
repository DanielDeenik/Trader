1 SimilarWeb — Data Collection Anatomy

First Principle: How Do You Know What 4 Billion People Do Online?

SimilarWeb's core insight is that you don't need to observe everyone — you need a representative panel combined with multiple independent data streams, then you model the rest. This is the same principle behind political polling, Nielsen ratings, and credit scoring.

The 5-Layer Data Collection Stack

Layer 1 — Contributory Network (Panel Data)

Browser extensions (Chrome, Firefox) installed by millions of users who opt in to share anonymized browsing data. This provides ground-truth clickstream: which URLs were visited, for how long, in what sequence. This is their highest-fidelity signal but smallest sample.

LATTICE parallel: Our StockTwits + Reddit connectors are panel-like — small, high-signal samples of sentiment.

Layer 2 — ISP Partnerships

Aggregated traffic data from Internet Service Providers worldwide. ISPs see DNS lookups and connection metadata (not page content). This gives volume estimates at massive scale but low granularity — you know "paypal.com got X visits" but not which pages or how long.

LATTICE parallel: Google Trends gives us volume without depth. Same trade-off.

Layer 3 — First-Party Analytics Integration

Millions of sites voluntarily share their analytics (via SDK integration) in exchange for SimilarWeb benchmarking data. This is a value exchange: you give your data, you get your competitors' data.

LATTICE parallel: We don't have this layer. Our web_traffic connector proxies SimilarWeb's output, not raw analytics.

Layer 4 — Public Web Crawling

Automated bots crawl millions of pages monthly, capturing page structure, links, content, meta tags, and technical signals. This provides the relationship graph: who links to whom, what content exists, how sites are structured.

LATTICE parallel: Our Wikipedia + YouTube connectors are crawl-like — scraping public pages for structured data.

Layer 5 — Machine Learning Synthesis

All four layers feed into ML models that extrapolate from samples to population estimates. The panel provides calibration data; the ISP data provides scale; the first-party data provides ground truth for validation; the crawl data provides structure. The model outputs the final numbers.

LATTICE parallel: Our aggregate_signal calculation in data_sources.py is a primitive version of this — weighted averaging across sources.

Key Insight From SimilarWeb

SimilarWeb's power comes from triangulation — no single data source is trusted alone. Every metric is cross-validated by at least 2 independent collection methods. Their ML layer doesn't just average; it weights sources by their reliability for each specific metric. This is the principle LATTICE must adopt: confidence scores per source per metric, not flat averages.

2 TickerTrends — Workflow Anatomy

First Principle: How Do You Turn Consumer Behavior Into Stock Predictions?

TickerTrends' insight is that consumer behavior data (search, social, app usage) is a leading indicator of revenue, and revenue is what drives stock prices. If you can measure the leading indicator in real-time, you have a 4-12 week information advantage over Wall Street, which waits for quarterly earnings.

Phase 1: Discover — Exploding Trends

A real-time dashboard surfacing brands, products, and hashtags with exponential growth across search + social + app metrics. Velocity is king here: day-over-day, week-over-week, month-over-month acceleration. The user doesn't need to know what to look for — the system surfaces it.

LATTICE gap: We have no discovery/scan-all view that surfaces rising signals autonomously. The daily-lattice-scan task is backend-only. The dashboard requires manual ticker input.

Phase 2: Analyze — The "Google Search of Alt Data"

Type a ticker, get a unified view across 8-12 data sources. Each source rendered as its own panel with trend visualization. AI-generated summaries of the last 6 earnings calls. Cluster charts showing semantically related keywords. Historical financials for context.

LATTICE match: Our single-ticker scan view does this. 10 source cards with sparklines + metrics. But we lack earnings call summaries and keyword clustering.

Phase 3: Score — Social Arbitrage + Investor Saturation

Two proprietary composite scores. Social Arbitrage Score (0-100): how strong is the multi-source signal? Investor Saturation Score (0-100): how many institutional investors already know? High signal + low saturation = maximum alpha. This is the Camillo framework operationalized.

LATTICE match: Our Mosaic Gauge (aggregate_signal) maps to Social Arbitrage Score. We lack an Investor Saturation equivalent — could derive from StockTwits follower growth + analyst coverage count.

Phase 4: Predict — KPI Forecasting

ML models trained on historical alternative data vs. reported outcomes. Predicts next-quarter KPIs (revenue, users, bookings) 4-12 weeks early. Red dots on the dashboard show forecast values. This is where alternative data graduates from "interesting" to "actionable."

LATTICE gap: We have no predictive layer. Our divergence detection is descriptive ("price and traffic are diverging") not predictive ("next quarter revenue will beat by X%").

Phase 5: Manage — Consumer Interest Tracker

Weekly composite metric blending search + traffic + app + social into a single number per ticker. This is the "check once a week" metric that tells you if your thesis is on track without drowning in data. Updates every 7 days, not every 90.

LATTICE match: Our daily scan captures this data. But we don't distill it into a single trackable weekly metric with historical trend. The aggregate_signal is per-scan, not longitudinal.

3 Gap Analysis: What Neither Platform Does (LATTICE's Edge)

SimilarWeb's Blindspot

SimilarWeb sees traffic but not intent. It tells you PayPal.com got 485M visits, but not whether those visitors were bullish or bearish. It has no connection to financial markets, no sentiment layer, no investment framework.

TickerTrends' Blindspot

TickerTrends sees signals but not relationships. It scores each ticker independently. It doesn't model cross-asset connections: that SOFI + MA + FOUR + PAYO all announcing stablecoin initiatives is a correlated signal, not four independent ones.

LATTICE's Opportunity

LATTICE can model the relationships between signals, not just the signals themselves. The knowledge graph is the structural advantage: it captures that a Google Trends spike for "BNPL" connects to AFRM, PYPL, SQ, and FOUR simultaneously, and that this cluster is gaining momentum.

4 LATTICE Knowledge Graph Architecture

First Principle: Why a Graph?

Spreadsheets and dashboards show you what is happening. Graphs show you why and what connects to what. When FOUR, PAYO, SOFI, and PYPL all announce stablecoin settlement in the same month, a spreadsheet shows four independent events. A knowledge graph shows one phenomenon with four manifestations — and that phenomenon (stablecoin-as-payment-rail) is the actual investable thesis.

The Graph's Unique Power: Cross-Asset Signal Correlation

When Google Trends shows "stablecoin payments" spiking, the graph doesn't just tag one ticker — it propagates the signal to every connected node: SOFI (via SoFiUSD), FOUR (via stablecoin settlement), PAYO (via stablecoin bank), PYPL (via PYPL USD), and MA (via SoFiUSD partnership). The CORRELATES_WITH and BELONGS_TO edges are what make this possible. TickerTrends would show you five independent trend lines. LATTICE shows you one theme with five exposure points.

Stage 1: Raw Collection (SimilarWeb-Style Multi-Source)

Each connector in data_sources.py fetches raw data from a public API or web endpoint. The connector's job is normalization: convert source-specific formats into the universal DataPoint dataclass with source, ticker, metric, value, previous_value, delta_pct, timestamp, confidence, and tags.

This mirrors SimilarWeb's Layer 4 (crawling) and Layer 1 (panel data). We're collecting from diverse, independent streams. The key principle from SimilarWeb: diversity of sources matters more than depth of any single source.

Stage 2: Per-Source Aggregation (TickerTrends-Style Scoring)

Multiple DataPoints from the same source for the same ticker get rolled up into a SourceSnapshot with a signal_strength score. This is analogous to TickerTrends' per-source trend visualization — each data source gets its own card showing velocity, direction, and confidence.

The signal_strength ranges 0-10 and considers: raw metric values, delta percentages, confidence of the data, and source-specific weighting from the DB.

Stage 3: Cross-Source Synthesis (The LATTICE Advantage)

All SourceSnapshots for a ticker get assembled into a MultiSourceScan. This is where the Camillo Mosaic Theory kicks in. The aggregate_signal is a weighted combination across all sources. But more importantly, _detect_divergences() runs pairwise comparisons to find mismatches — the alpha signals.

Current divergence types detected: volume/price, sentiment/price, dev_activity/price, attention/price, traffic/price. Each one is a potential graph edge.

Stage 4: Graph Construction (What Makes LATTICE a Lattice)

This is where LATTICE transcends both SimilarWeb and TickerTrends. Each scan result generates graph operations:

  • Upsert Ticker node with latest metrics (price, mcap, volume)
  • Create Signal nodes for each notable data point (search spike, sentiment shift, traffic surge)
  • Create HAS_SIGNAL edges connecting tickers to their signals
  • Create DIVERGES_FROM edges for each detected divergence
  • Update BELONGS_TO edges when theme membership changes
  • Compute CORRELATES_WITH edges by comparing signal patterns across tickers in the same sector

Over time, the graph accumulates history. A single scan is a data point. A month of daily scans is a thesis.

Stage 5: Theme Emergence (Autonomous Discovery)

Themes are not hardcoded — they emerge from graph analysis. When multiple tickers in the same sector show correlated signals (e.g., SOFI + FOUR + PAYO all show "stablecoin" keyword spikes within the same 2-week window), the graph's clustering algorithm should surface this as a nascent theme. The 6 themes currently in the dashboard are manually identified; the graph should identify them programmatically.

This is the Gold Rush Lifecycle: emerging (1-2 tickers showing signal) → validating (3+ tickers, cross-source confirmation) → confirmed (sector-wide, analyst coverage begins) → saturated (consensus priced in, TickerTrends Investor Saturation Score equivalent).

Layer 1: Discovery View (TickerTrends "Exploding Trends" Pattern)

What the graph provides: Query all Signal nodes created in the last 7 days, ranked by delta_pct. Surface tickers with the highest-velocity positive signals that don't yet have high aggregate scores (i.e., the market hasn't priced them in). This is the "emerging" ring of the Gold Rush.

UI elements: Rising signal feed (real-time list), velocity sparklines, "New This Week" badges, threshold-based alerts.

Currently missing from lattice_research.html. Would be a new top-level tab alongside "Payments."

Layer 2: Universe View (SimilarWeb "Competitive Benchmark" Pattern)

What the graph provides: Query all Ticker nodes in a sector, join with their latest Metric nodes (price, mcap, YTD) and aggregate_signal from the most recent scan. Group by subsector. Rank by signal strength or upside.

UI elements: Sortable table (implemented ✓), subsector pills (implemented ✓), mosaic themes (implemented ✓), divergence signals (implemented ✓).

This is what we just built into the dashboard. It shows the "Sector Overview" that SimilarWeb provides for competitive benchmarking.

Layer 3: Ticker View (TickerTrends "Analyze" Pattern)

What the graph provides: Traverse from the Ticker node through HAS_SIGNAL edges to all Signal nodes, through MEASURED_BY to Source nodes, and through DIVERGES_FROM to find divergence pairs. The 8-Atom scoring comes from evaluating signals against the Camillo criteria.

UI elements: Data Source Grid (10 cards with sparklines — implemented ✓), Camillo 8-Atom Framework (implemented ✓), Divergence Alerts (implemented ✓), Thesis Builder (implemented ✓), Sector Comparison Bars (implemented ✓).

Layer 4: Graph View (LATTICE's Unique Contribution)

What the graph provides: The raw graph itself — nodes and edges visualized as a force-directed network. Ticker nodes sized by signal strength. Theme nodes as cluster centers. Edge opacity by correlation strength. This is the "lattice" that gives the platform its name.

UI elements: Interactive force-directed graph (currently mock ✓). Click a node → navigate to its Ticker View. Click a Theme node → see all connected tickers. Edge tooltips show relationship type and strength.

Graph queries that power insights:

  • "Which tickers are most connected to the stablecoin theme?" → traverse BELONGS_TO edges from stablecoin_settlement theme node
  • "What signals are diverging for PYPL?" → traverse HAS_SIGNAL → filter DIVERGES_FROM edges
  • "Which tickers correlate most with FOUR?" → traverse CORRELATES_WITH edges, rank by weight
  • "Is this theme emerging or saturated?" → check LIFECYCLE_STAGE edge, count connected tickers, measure signal velocity

Proposed Graph Storage Schema (SQLite)

-- Nodes
CREATE TABLE graph_nodes (
  id TEXT PRIMARY KEY,     -- e.g., "ticker:PYPL", "theme:stablecoin_rail"
  type TEXT NOT NULL,      -- ticker | signal | metric | theme | source
  properties TEXT,         -- JSON blob
  created_at TEXT,
  updated_at TEXT
);

-- Edges
CREATE TABLE graph_edges (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  from_node TEXT NOT NULL,
  to_node TEXT NOT NULL,
  edge_type TEXT NOT NULL,  -- HAS_SIGNAL | DIVERGES_FROM | BELONGS_TO | etc.
  weight REAL DEFAULT 1.0,
  properties TEXT,          -- JSON blob (timestamp, confidence, scan_id)
  created_at TEXT,
  FOREIGN KEY (from_node) REFERENCES graph_nodes(id),
  FOREIGN KEY (to_node) REFERENCES graph_nodes(id)
);

-- Indexes for graph traversal
CREATE INDEX idx_edges_from ON graph_edges(from_node);
CREATE INDEX idx_edges_to ON graph_edges(to_node);
CREATE INDEX idx_edges_type ON graph_edges(edge_type);
CREATE INDEX idx_nodes_type ON graph_nodes(type);

Build Priority (First Principles Order)

Priority 1 — Graph Storage

Add graph_nodes + graph_edges tables to the existing SQLite DB. Modify daily scan to write nodes/edges after each MultiSourceScan completes. This is the foundation everything else builds on.

Priority 2 — Cross-Ticker Correlation

After 7+ days of daily scans, compute CORRELATES_WITH edges by comparing signal time-series across tickers in the same sector. This enables "which tickers move together?" queries.

Priority 3 — Theme Detection

Cluster correlated tickers + shared keyword signals into Theme nodes. Use simple co-occurrence: if 3+ tickers in a sector share a signal keyword within 14 days, create a theme. This replaces the manual themes.

Priority 4 — Discovery Feed

Query the graph for signals with high delta_pct but low aggregate_signal (early stage, not yet consensus). Surface these as "Exploding Signals" in the dashboard. This is the TickerTrends "Discover" equivalent.
